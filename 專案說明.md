## 一、這個專案是什麼?

一個**工廠設備監控系統**。記錄每台機器「什麼時候在運轉、什麼時候停機、為什麼停」,然後自動算出:

- **稼動率(Utilization)** — 這台機有多少時間在真正運轉
- **停機原因 Pareto** — 停機主要是因為什麼(換線?故障?待料?)
- **良率(Yield)** — 做出來的東西有多少是良品

打開 `http://localhost:5243/` 就能看到一個監控儀表板(dashboard)。

## 二、整個系統分三塊(最重要的觀念)

像點餐:**你(前端)→ 服務生(後端)→ 廚房(資料庫)**。

```
┌─────────────┐   要資料    ┌─────────────┐   查資料   ┌─────────────┐
│   前端網頁   │ ─────────→ │   C# 後端    │ ────────→ │  SQL Server │
│ index.html  │            │ Controllers │  (跑 SQL)  │   (資料庫)   │
│  (你看到的) │ ←───────── │  (API)      │ ←──────── │  (真實資料)  │
└─────────────┘   回 JSON   └─────────────┘   回資料   └─────────────┘
```

| 角色 | 是什麼 | 負責 | 懂不懂 SQL? |
|------|--------|------|------------|
| **前端** | `wwwroot/index.html` | 顯示畫面、跟後端要資料 | ❌ 不用懂 |
| **後端** | `Controllers/*.cs`(C#) | 收請求、跑 SQL、回 JSON | ✅ 核心 |
| **資料庫** | SQL Server `MesMonitoring` | 存真實資料 | — |

## 三、一次完整的資料流(以「稼動率」為例)

1. 你打開網頁,JavaScript 用 `fetch('/api/dashboard/utilization')` **去敲後端的門**
2. 後端 `DashboardController.cs` 的 `[HttpGet("utilization")]` 這扇門收到請求
3. 後端跑裡面那段手寫 **SQL**(`SELECT ... GROUP BY`),去 SQL Server 查資料
4. SQL Server 回傳結果,後端包成 **JSON**:`[{"code":"M-001","utilization":86.7}]`
5. 前端拿到 JSON，用 `.map()` **一筆筆填進表格** → 你看到畫面

> 前後端之間的「共同語言」= **網址(API) + JSON**。

## 五、五張資料表

```
Workstations (工站)
    │ 1對多
    ▼
Machines (機台) ──────────┐
    │ 1對多               │ 1對多
    ▼                     ▼
StatusLogs (狀態記錄)   ProductionLogs (產量)
    │ 多對1
    ▼
DowntimeReasons (停機原因)
```

| 表 | 角色 | 重點 |
|----|------|------|
| `Workstations` | 工站(最上層) | |
| `Machines` | 機台,屬於某工站 | 外鍵 `WorkstationId` |
| **`StatusLogs`** | **狀態時間區間 — 系統心臟** | 稼動率全靠它算,存「區間」不是「當下狀態」 |
| `DowntimeReasons` | 停機原因對照表 | 正規化,避免自由打字 |
| `ProductionLogs` | 產量(良品/不良) | 算良率 |

## 八、面試三個必背重點

1. **StatusLog 存「時間區間」不存「當下狀態」** → 管理者要的是稼動率分析，全靠歷史區間用 `DATEDIFF` + `GROUP BY` 算出來。
2. **停機原因獨立成對照表(正規化)** → 用 `ReasonId` 關聯，現場原因要能統計分類，不能讓人自由打字。
3. **稼動率怎麼算** → 運轉時數 ÷ 總時數，在 `DashboardController` 用手寫 SQL 的 `GROUP BY` 完成。

> 殺手鐧:8 年製造背景 + 懂現場 + 會手寫 SQL → 「懂現場 + 會系統」的跨域組合。
